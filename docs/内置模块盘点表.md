# 内置模块盘点表（阶段 0 产物）

> 来源：直接读取 `world-engine-rules-loader.js` / `world-engine-evolution.js` / `world-engine-ui.js` / `world-engine-core.js` / `world-engine-presets.js`。
> 作用：自由模式抽象层的事实依据。每个内置模块在引擎里到底用了哪些「特权机制」，全列在这里——通用机制引擎（阶段 2）必须能覆盖这张表里出现的每一种机制。

## 一、模块总表

| # | moduleId | 中文名 | 规则正文(RULES) | 输出字段 | 容器 | 是否有专属机制 | 渲染函数(ui.js) |
|---|----------|--------|----------------|----------|------|----------------|-----------------|
| 1 | `world` | 世界运转 | ✅ | 无（仅规则） | none | 否（背景规则） | 经 `renderWorldCore`（世界摘要+稳定度，特殊顶层字段） |
| 2 | `events` | 事件链 | ✅ | `events` | array | ✅ 骰子+阶段机+清理 | `renderEventList` / `renderEventEditor` |
| 3 | `factions` | 势力 | ✅ | `factions` | array | ✅ 裁决 | `renderFactionList` / `renderFactionEditor` |
| 4 | `winds` | 风声 | ✅ | `winds` | array | ✅ 消散骰 | `renderWindList` / `renderWindEditor` |
| 5 | `influence` | 影响链 | ✅ | `influenceChain` | array | ✅ 生命周期 | `renderInfluenceChain` |
| 6 | `contact` | 主动接触与信息传播 | ✅ | 无（仅规则） | none | 否（背景规则） | 无面板 |
| 7 | `reputation` | 声誉 | ✅ | `reputation` | object | ✅ 裁决(4维×5档) | `renderReputation` |
| 8 | `economy` | 经济 | ✅ | `economy` | object | ✅ 裁决(climate)+上限 | `renderEconomy` / `renderEconomyEditor` |
| 9 | `enemies` | 仇敌录 | ✅ | `enemies` | array | ✅ 状态校验+清理+上限 | `renderEnemies` / `renderEnemyEditor` |
| 10 | `regional` | 区域突发事件 | ✅ | `regionalIncident` | object | ✅ 骰子(触发/冷却/持续) | `renderRegionalIncident` / `renderRIEditor` |
| 11 | `blackbox` | 信息黑盒 | ✅ | `blackbox` | object | ✅ 私密规则+上限 | `renderBlackbox` / `renderSecretEditor` |
| 12 | `trends` | 天下大势 | ✅ | `worldTrends` | array | ✅ 清理(无骰子) | `renderWorldTrends` / `renderTrendEditor` |

要点：
- **两类内置模块**：`world` 和 `contact` 是**纯规则模块**（只注入规则、不产出 JSON 字段、无面板）。其余 10 个是**输出模块**。→ 描述符规格必须支持 `container: 'none'`。
- 输出字段名映射在两处：`rules-loader.js: MODULE_FIELD_MAP` 与 `evolution.js: MODULE_FIELD_MAP`（**重复定义，阶段 1 需合并为单一来源**）。
- 输出 schema 契约在 `rules-loader.js: OUTPUT_SCHEMAS`，并经 `mergeOutputSchema` 叠加预设 `schemaOverrides`（addFields / overrideFields / hiddenFields）——这套合并逻辑已通用，可直接复用到自定义模块。

## 二、机制盘点（通用机制引擎的需求来源）

### A. 骰子（DICE）——注意：不是一种，是三种不同骰
| 模块 | 骰子逻辑 | 关键参数/常量 | 代码位置 |
|------|----------|---------------|----------|
| `events` | 阶段推进骰：`dice(1-100) > threshold` 则进度+1；`threshold = stageBase - 200·r·(1-r) + levelAdjust`，`r=stageRound/9` | `EVENT_STAGE_BASE`（冲突 95/85/75，推进 75/85/95） | `evolution.js` `forceTriggerEvents` (~344) |
| `regional` | 触发骰：`dice = random()`，`dice < chance` 触发；含冷却(cooldown)与持续(duration)；类型用 `weightedPick` 加权抽 | `getRegionalConfig`（chance/duration/cooldown/types，来自预设 `regionalIncidents`） | `evolution.js` `rollRegionalIncident` (~197) |
| `winds` | 消散骰：按类型查 `WIND_DECAY` 算消散概率，`dice ≤ chance` 则该风声消散 | `WIND_DECAY`（announcement/report/rumor/sentiment 各一组 base/grace/linear/quadratic） | `evolution.js` `decayWinds` (~433) |

> 结论：`DiceEngine` 需要覆盖「阈值推进骰 / 概率触发骰(带冷却+持续+加权类型) / 概率消散骰」三个家族。这是阶段 2 工作量最大的一块。

### B. 阶段机（STAGE MACHINE）
| 模块 | 状态序列 | 终局态 | 代码位置 |
|------|----------|--------|----------|
| `events` (conflict) | 萌芽 → 发酵 → 逼近 → 已爆发 | 已爆发 / 已消散 | `EVENT_STAGE_ORDER` / `EVENT_FINAL_STAGE` / `EVENT_TERMINAL_STAGES` / `advanceStageRound` |
| `events` (progress) | 筹备 → 执行 → 关键 → 已完成 | 已完成 / 已失败 | 同上 |

> 每阶段内有 `stageRound`（1-9）进度条；≥9 自动晋级；终局锁 9/9。`StageMachine` 需支持「多 type 各一套序列 + 阶段内进度 + 终局锁定」。

### C. 裁决（VERDICTS：枚举 → 描述文字）
| 模块 | 轴 | 固定档位（INTERNAL_SCHEMA，presets.js） |
|------|-----|------------------------------------------|
| `reputation` | authority / common / shadow / circuit（4维） | 天怒人怨 / 声名狼藉 / 默默无闻 / 受人尊敬 / 万众敬仰（5档） |
| `factions` | status | 鼎盛 / 稳固 / 倾轧 / 困顿 / 衰落 / 瓦解（6档） |
| `factions` | relation | 血盟 / 盟友 / 友好 / 中立 / 冷淡 / 敌对 / 世仇（7档） |
| `economy` | climate | 繁荣 / 平稳 / 衰退 / 动荡（4档） |
| （稳定度，顶层非模块） | stability | 天下太平 / 暗流浮动 / 局势紧张 / 动荡失序 / 崩坏边缘（5档） |

> 档位 key 在引擎内固定，预设只改每档的「描述文字」(verdicts)。`VerdictEngine` 需支持「多轴 × 固定档位 × 可换描述」，且要**兼容**这些既有固定枚举（classic 不可回归）。

### D. 生命周期 / 清理 / 上限（LIFECYCLE）
| 模块 | 规则 | 代码位置 |
|------|------|----------|
| `events` | 负面终局(已消散/已失败)下轮即删；正面终局(已爆发/已完成)保留 `2+level*2` 轮 | evolve 清理段 (~948) |
| `influence` | 创建 8 轮后过期；列表上限 12 | evolve (~898) |
| `enemies` | `已终结` 保留 20 轮；上限 8 | evolve (~860) |
| `trends` | `status==='已结束'` 即删；不参与骰子 | evolve (~967) |
| `economy.signals` | 上限 8 | evolve (~915) |
| `blackbox` | secretActions+secretAssets 合计上限 12 | evolve (~922) |

### E. 合并（MERGE：API 返回 → state）
| 形态 | 模块 | 合并键 | 实现 |
|------|------|--------|------|
| array 按键合并 | events(name) / factions(name) / winds(topic) / enemies(name) / influence(trigger) | 同键更新、异键新增 | `core.addEvent/addFaction/addWind` + evolve 内联 |
| object 浅合并 | economy / reputation | `Object.assign` | evolve (~875) |
| object 特殊合并 | regional | `mergeRegionalIncident`（受骰子门控，不接受 API 自发） | evolution.js (~287) |
| object 整体替换 | blackbox | 直接替换 + 上限裁剪 | evolve (~922) |

> `Merge` 通用器需支持：array-by-key（键名可配）/ object-assign / object-replace。`regional` 的「骰子门控合并」是特例，归到 DiceEngine 联动。

## 三、对抽象层的直接结论
1. 描述符必须支持 `container ∈ {none, scalar, object, array}`。
2. `mechanics` 至少含四类：`dice`(三家族) / `stages` / `verdicts` / `lifecycle(retention+cap)`；外加 `merge` 策略字段。
3. `world`/`contact` 证明「纯规则模块」是合法且必要的形态。
4. `OUTPUT_SCHEMAS + mergeOutputSchema` 与 `renderSchemaExtraFields` 已是通用件，自定义模块可直接复用，**不必从零造轮子**。
5. 固定枚举（5/6/7/4/5 档）是 classic 灵魂，`VerdictEngine` 以「兼容」为第一目标。
