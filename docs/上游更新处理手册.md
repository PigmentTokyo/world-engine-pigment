# 上游更新处理手册（PigmentTokyo/World 改版维护）

> 原版（上游）= `DlSNlGHT/World`；本仓库 = 你的重度改版 `PigmentTokyo/World`。
> 两边代码已大幅分叉，**不能无脑 merge / rebase / cherry-pick**。
> 正确姿势：**拉取 → 评估 → 挑拣 → 手工移植 → 测试 → 提交**。

---

## 0. 一次性准备（只需做一次）

确认上游远程已配置：

```powershell
git remote -v
```

应能看到：

```
origin    https://github.com/PigmentTokyo/World.git
upstream  https://github.com/DlSNlGHT/World.git
```

如果没有 `upstream`，加上：

```powershell
git remote add upstream https://github.com/DlSNlGHT/World.git
```

## 1. 拉取上游最新代码

```powershell
git fetch upstream
```

这只更新本地的 `upstream/master` 引用，**不会改动你的任何文件**，放心拉。

## 2. 看上游新增了哪些提交

**上次对齐点**记录在本手册末尾「同步记录」。把它填进下面命令的 `<上次同步点>`：

```powershell
# 只看非合并提交，最旧的在上面
git log --oneline --reverse --no-merges <上次同步点>..upstream/master
```

例如当前记录是 `021dc21`：

```powershell
git log --oneline --reverse --no-merges 021dc21..upstream/master
```

也可以直接去 GitHub 看：`https://github.com/DlSNlGHT/World/commits` 和它的 changelog / release 说明。

---

## 3. 分类：哪些值得看

按提交前缀和内容分三类，**只重点处理前两类**：

| 类别 | 关键词 | 处理 |
|---|---|---|
| 修 bug / 性能 | `fix(...)`、`perf`、"修复"、"卡死"、"丢失"、"崩溃" | **重点**，优先评估 |
| 新功能 | `feat(...)`、"新增" | 按需，看你要不要这个能力 |
| 杂项 | `chore`、`docs`、版本号、changelog | 基本忽略 |

---

## 4. 逐个评估候选提交

对每个想移植的提交，先看它动了哪些文件、改了什么：

```powershell
git show <sha> --stat      # 改了哪些文件
git show <sha>             # 完整 diff
```

然后用**三个问题**判断能不能直接用：

1. **你 fork 是否已经有等价实现？**
   先 grep 关键函数名 / 符号，别重复移植。
   ```powershell
   git grep -n "某个函数名或关键字"
   ```

2. **它改的文件 / 模块，你 fork 里存在吗？**
   你和上游虽然都是 `world-engine-*.js` 模块化，但**不完全一样**。
   （详见第 6 节「已知差异地图」。）

3. **会不会和你的改造正面冲突？**
   比如上游的「引擎预设系统」和你自己的预设系统是两套架构，硬移会打架。

三问都过 → 可移植；任意一条不过 → 跳过或只取其中不冲突的部分。

## 5. 手工移植（不要 cherry-pick）

因为文件上下文对不上，`git cherry-pick` 几乎必冲突。正确做法：

1. 读懂上游那段 diff 的**逻辑意图**（它在防什么 bug / 优化什么）。
2. 在你 fork 的**对应位置**手动改成等价逻辑（变量名、调用方式以你 fork 为准）。
3. 改完**立刻自检**（见第 7 节）。

> 经验：一个 bug 往往是上游好几个小提交叠出来的最终形态。先把这些小提交**顺着看一遍**，照**最终状态**移植，别一个一个搬中间过程。

## 6. 已知差异地图（省时间，先看这个）

这是上次（对齐到 `v2.1.1` / `021dc21`）实操摸清的结论，**下次仍大概率成立**：

- **你 fork 没有** `world-engine-chatcache.js`。上游的「酒馆缓存」模块在你这边由 `world-engine-store.js` 直接把 chat-scoped 数据镜像进 `chat_metadata` 实现了**跨设备同步**。所以上游 chatcache 相关提交基本 **N/A**。
- **预设系统是两套**：你是 `world-engine-presets.js` + `world-engine-preset-ui.js`；上游是 `world-engine-preset.js` + `WORLD_ENGINE_PRESET.getSegmentOverride` 那套。上游 preset 相关的 `feat/fix` **不能直接用**，符号都对不上。
- **你已经有的**（无需再移植，先 grep 确认）：正则过滤、世界书蓝绿灯触发（`worldbookTrigger`）、酒馆代理绕 CORS、诊断包导出、设置页选项卡。
- **自动推演锚点 / 计数逻辑**：你 fork 的 `world-engine.js` 已是上游修复后的「超集」（用 `getChatLayer` + `restoreCheckpoint(cp.chatLayer!=null)` + fingerprint 兜底，且**没有**上游那个有 bug 的「2X 重置」块）。上游这一串锚点 `fix` 多半已等价存在。
- **存档备份**：你独有 `world-engine-backup.js`（本地滚动备份，上游没有）。移植上游时别被它覆盖。
- **模块加载**：新模块要加进 `world-engine.js` 顶部的 `MODULES` 数组（注意顺序：依赖 `core` 的放 core 之后，要被 `ui` 引用的放 `ui` 之前）。`manifest.json` 只挂 `world-engine.js` 一个入口，其余靠它动态加载。

## 7. 自检 + 测试（每次移植后必做）

```powershell
# 1) 改过的每个文件语法过一遍
node --check world-engine-某文件.js

# 2) 跑全部单测，必须全绿
Get-ChildItem tests\*.test.js | ForEach-Object { node $_.FullName }
```

测试里有三道**回归红线**（classic 基线零差异、动态闸门零差异、UI 渲染零差异），只要它们过了，说明没破坏核心行为。新功能最好补一个对应的 `tests\xxx.test.js`。

## 9. 本机踩过的坑（遇到了照这里修）

- **`fatal: Unable to create ... index.lock: File exists`**
  残留锁文件。确认没开 git 图形客户端后删除：
  ```powershell
  Remove-Item .git\index.lock -Force
  ```

- **`bad config line 1 in file .git/config`**
  config 被写坏（曾被填成 null 字节）。用文本编辑器把 `.git/config` 重建为：
  ```ini
  [core]
      repositoryformatversion = 0
      filemode = false
      logallrefupdates = true
  [remote "origin"]
      url = https://github.com/PigmentTokyo/World.git
      fetch = +refs/heads/*:refs/remotes/origin/*
  [remote "upstream"]
      url = https://github.com/DlSNlGHT/World.git
      fetch = +refs/heads/*:refs/remotes/upstream/*
  [branch "master"]
      remote = origin
      merge = refs/heads/master
  ```

- **大文件改完结尾被截断 / 莫名语法错**
  改完一定 `node --check`，并看文件结尾是否完整（应以 `})();` 之类正常收尾）。被截断就从干净版本重写整个文件，别在坏文件上接着补。

- **行尾**：仓库 `.gitattributes` 强制 LF。保存文件保持 LF，别让编辑器换成 CRLF，否则 diff 会炸成整文件变更。
