# 模块描述符规格 ModuleDescriptor（阶段 0 产物）

> 这是自由模式的核心数据契约。整个引擎（规则注入 / 推演输出契约 / 解析合并 / 渲染 / 机制）都将围绕这一个对象展开。
> 设计目标：**内置 12 模块能被它无损包裹（classic 零回归），自定义模块也用同一套字段描述。**

## 0. 设计取舍（已确认）
- 通用模块**保留可配置机制**（dice / stages / verdicts 都做成 config 驱动）。
- classic 与 free **允许混合**：自由预设可按 `id` 引用内置模块，复用其机制。
- 一个引擎 + 模式开关，不做平行双引擎。

## 1. 顶层结构 ModuleDescriptor

```js
{
  id: 'cultivation',          // 唯一标识；也用作 state 字段名（自定义模块）。必须 /^[a-zA-Z][a-zA-Z0-9_]*$/，且不与内置字段冲突
  name: '模块十三：修为',       // 面板/开关显示名。保留“模块X：”前缀是惯例，非强制
  enabled: true,              // 等价于现在的 disabledModules（关 = false）
  kind: 'custom',             // 'builtin' | 'custom'
  order: 13,                  // 排序（混合模式下内置与自定义统一排序）
  rules: '……发给 AI 的推演指引原文……',  // 对应内置的 RULES.content

  container: 'array',         // 'none' | 'scalar' | 'object' | 'array'
  itemKey: 'name',            // container=array 时的合并主键（默认 'name'）
  fields: { /* 见 §2 */ },    // container=object/array 时的字段 schema
  scalarType: 'string',       // container=scalar 时的值类型

  display: { /* 见 §3 */ },
  mechanics: { /* 见 §4 */ }
}
```

- `kind:'builtin'`：`rules/fields/mechanics/display` 可省略，引擎从内置实现取（包裹器模式）。预设里引用内置模块时通常只写 `{ id, kind:'builtin', enabled, order, name? }`。
- `container:'none'`：纯规则模块（如 `world`/`contact`），忽略 `fields/display/mechanics`，只注入 `rules`。

## 2. fields（字段 schema）
直接沿用现有 `OUTPUT_SCHEMAS[*].fields` 的形状，零新增概念：

```js
fields: {
  realm:     { type:'enum', enum:['炼气','筑基','金丹','元婴'], description:'当前境界', example:'筑基', display:true },
  spiritQi:  { type:'number', description:'灵气值 0-100', example:42, display:true },
  technique: { type:'string', description:'主修功法', example:'青冥诀', display:false }
}
```
- `type`：`string|number|boolean|enum|array<string>|object|array<object>`（与现有一致）。
- `array<object>` 用 `itemFields` 描述子字段（同现有）。
- `display`：是否在主面板卡片展示。
- 预设级覆盖仍走现有 `schemaOverrides`（addFields/overrideFields/hiddenFields）+ `mergeOutputSchema`，无需新机制。

## 3. display（渲染配置）
驱动通用卡片渲染器（替代逐模块手写的 `render<Module>`）：

```js
display: {
  style: 'cards',          // 'cards' | 'table' | 'keyvalue' | 'list'
  titleField: 'name',      // 卡片标题取哪个字段
  badgeFields: ['realm'],  // 顶部徽标字段
  subtitleField: 'scope',  // 副标题
  bodyFields: ['technique','spiritQi'],  // 卡片正文展示字段
  emptyText: '暂无修为记录',
  paged: true, perPage: 4  // 复用现有 renderPagedList
}
```
- 容器 array → `cards`/`table`/`list`；object → `keyvalue`/`cards(单卡)`。
- 内置模块 `kind:'builtin'` 继续用其专属渲染函数，不受 display 影响（兼容优先）。

## 4. mechanics（可配置机制）——核心
盘点表已证明机制有四类，规格如下。**未声明的机制即关闭**（纯字段模块）。

### 4.1 dice（三家族，按 mode 区分）
```js
dice: {
  mode: 'threshold',   // 'threshold' | 'trigger' | 'decay'
  // —— threshold（阶段推进骰，参照 events）——
  base: { '萌芽':95,'发酵':85,'逼近':75 }, curve:'parabolic', levelField:'level',
  // —— trigger（触发骰，参照 regional）——
  chance:0.03, cooldownRounds:5, durationRounds:5, weightedTypes:[{type,weight,guide}],
  // —— decay（消散骰，参照 winds）——
  byTypeField:'type', table:{ rumor:{base:25,grace:1,linear:5,quadratic:3}, /*…*/ }
}
```

### 4.2 stages（阶段机，参照 events）
```js
stages: {
  typeField: 'type',                 // 多 type 各一套序列；省略则单序列
  order: { conflict:['萌芽','发酵','逼近'], progress:['筹备','执行','关键'] },
  finalStage: { conflict:'已爆发', progress:'已完成' },
  terminalStages: { conflict:['已爆发','已消散'], progress:['已完成','已失败'] },
  progressField: 'stageRound', progressMax: 9   // 阶段内进度 1-9，≥max 自动晋级
}
```

### 4.3 verdicts（裁决，参照 reputation/factions/economy）
```js
verdicts: {
  axes: {
    realm: { levels:['炼气','筑基','金丹','元婴'], text:{ '炼气':'根基初成…', /*…*/ } }
  }
}
```
- `levels` 固定、`text` 可换（与内置固定枚举同构）。内置 5/6/7/4 档枚举以 `kind:'builtin'` 形式注册，**保证兼容不替换**。

### 4.4 lifecycle（生命周期/清理/上限）
```js
lifecycle: {
  cap: 12,                                  // 列表上限
  expireRounds: 8,                          // 创建 N 轮后过期（influence）
  terminalField: 'status',                  // 终局判定字段
  terminalRemove: ['已结束','已失败'],        // 命中即下轮删
  terminalRetain: { field:'stage', values:['已爆发','已完成'], rounds:'2+level*2' } // 保留若干轮
}
```

### 4.5 merge（合并策略）
```js
merge: 'arrayByKey' | 'objectAssign' | 'objectReplace'  // 默认按 container 推断
```

## 5. 与现有结构的关系（迁移）
| 现状 | 自由模式 |
|------|----------|
| `RULES[i].content` | `descriptor.rules` |
| `RULES[i].comment` | `descriptor.name` |
| `OUTPUT_SCHEMAS[id]` | `descriptor.container/fields` |
| `MODULE_FIELD_MAP[id]` | `descriptor.id`（自定义）/ 内置保留映射 |
| `preset.disabledModules` | `descriptor.enabled=false`（保留旧字段做兼容迁移） |
| evolution 各专属机制 | `descriptor.mechanics` 的某个配置实例 |
| `render<Module>` | `descriptor.display` 驱动的通用渲染（内置仍走专属函数） |

## 6. 预设层落地
```js
preset = {
  …现有字段…,
  mode: 'classic' | 'free',      // 默认 classic
  modules: [ ModuleDescriptor… ] // 仅 free/混合时非空；classic 留空走内置
}
```
- `mode:'classic'`：`modules` 忽略，引擎用内置 12 模块 + disabledModules + schemaOverrides（**与现状完全一致**）。
- `mode:'free'`：引擎用 `modules`；其中 `kind:'builtin'` 项复用内置实现（混合）。

## 7. 不变量（Invariants · 红线）
1. 🔒 `mode:'classic'` 下，注入 prompt 与 state 演化必须与改造前**逐字/逐字段一致**。
2. 内置固定枚举（声誉5/势力6/关系7/经济4/稳定5）只可兼容、不可替换。
3. 自定义 `id` 不得与内置 state 字段冲突（保存时校验拦截）。
4. 机制未声明 = 关闭；纯字段模块必须能跑（只注入规则 + 通用合并 + 通用渲染）。
